Ideation Artifact

Combine pacto-download into the pacto-app repo

Date: 2026-07-11 Topic: download-site-integration Mode: repo-grounded
Focus: combine the pacto-download site into pacto-app and eliminate manual config.js updates

Grounding Context

Project shape. Pacto is a Tauri v2 desktop app: a SvelteKit/Svelte 5 static SPA (src/) backed by a Rust backend (src-tauri/). The build uses @sveltejs/adapter-static with a fallback to index.html and SSR disabled. The frontend is built by Vite and consumed by the Tauri CLI.

Auth layout. src/routes/+layout.svelte gates every route behind an auth check: unauthenticated users see <Login />. Any public download page must either bypass that layout or live outside the src/routes/ tree.

Release pipeline. .github/workflows/release.yaml triggers on v* tags, generates a changelog with git-cliff, and uses tauri-apps/tauri-action@v1 to build cross-platform bundles (macOS aarch64/x86_64, Linux x86_64 and optional aarch64, Windows x86_64) and publish a draft GitHub release with updater JSON.

Current download site. The separate pacto-download repo has a config.js that hardcodes releaseTag: "v0.1.0" and a fixed list of five filenames. It is updated manually per release.

External prior art. Tauri/Electron apps commonly keep binaries on GitHub Releases and serve a marketing/download page from a static host such as GitHub Pages, Cloudflare Pages, or Vercel. The same-repo pattern is well established (e.g., OpenIT uses an Astro landing site that fetches the latest GitHub release at build time). Best practice is to avoid storing binaries in git, avoid hardcoding versioned filenames, and rebuild the site when a release is published.

Ranked Ideas

Idea 1

SvelteKit public /download route with build-time GitHub API and GitHub Pages

Confidence 85% Complexity Medium Axis: Code placement
Description

Add a new src/routes/download/+page.svelte route that opts out of the auth-gated root layout via +layout@.svelte or a public route group such as (public). Prerender the page with the existing static adapter. At build time, fetch the latest release from the GitHub API using GITHUB_TOKEN and map the published asset names to per-platform download buttons. Deploy the built site to GitHub Pages from a workflow triggered on release:published.

Basis

direct: src/routes/+layout.svelte gates every route behind {#if $isAuthenticated && $currentUser}; svelte.config.js already uses @sveltejs/adapter-static with fallback: "index.html". external: OpenIT's same-repo landing/ Astro site fetches the latest GitHub release at build time and links to platform assets.

Rationale

This collapses the second codebase into the same repo and build pipeline while removing the manual per-release edit of config.js. It keeps the download page on the same design system as the app and lets future marketing routes grow naturally.

Downsides

Requires carving a public route out of an auth-gated SvelteKit app; mixing app and marketing surfaces may complicate layout boundaries and bundle splitting; prerendering needs asset data available at build time.

Idea 2

Separate landing/ static site with generated manifest and GitHub Pages

Confidence 85% Complexity Medium Axis: Code placement
Description

Add a lightweight static site at the repo root (e.g., landing/ or download/) using Vite, Astro, or plain HTML. Keep it outside the SvelteKit route tree so the auth-gated layout is never involved. Extend the release workflow to generate a pacto-release.json manifest mapping platforms to the actual asset URLs published by tauri-action. Consume the manifest at build time and deploy the site to GitHub Pages.

Basis

direct: src/routes/+layout.svelte renders <Login /> for unauthenticated users; release.yaml already produces a latest.json via tauri-action. reasoned: A separate directory avoids risky root-layout refactoring and lets marketing surfaces evolve independently from the Tauri desktop app.

Rationale

This gives the same single-source-of-truth benefit while decoupling the public marketing site from the auth-gated app. The generated manifest becomes a reusable artifact for future distribution channels such as package managers or mirroring scripts.

Downsides

Adds a second build tool and a small amount of duplicated CSS/branding logic; requires a second CI deploy step; the public site and the app must be kept visually consistent.

Idea 3

Minimal static/download.html with client-side GitHub API fetch

Confidence 65% Complexity Low Axis: Code placement
Description

Drop a single static file into static/download.html (plus a small JS file) that calls the GitHub Releases API from the browser and renders platform download cards. No new SvelteKit route, no new build pipeline, and no changes to the release workflow beyond copying the file to the host.

Basis

direct: static/ currently holds only images and JS helpers; no public pages exist yet. external: The GitHub Releases API endpoint is https://api.github.com/repos/covenant-gov/pacto-app/releases/latest.

Rationale

This is the smallest possible integration that removes the separate repo and kills the manual per-release PR. It avoids touching the SvelteKit routing, auth layout, or build pipeline entirely.

Downsides

Unauthenticated GitHub API calls are capped at 60 requests per hour per IP; users behind NATs or VPNs may hit the limit; the page is not SEO-friendly; it breaks without JavaScript.

Idea 4

Reuse the Tauri updater latest.json as the data source

Confidence 80% Complexity Medium Axis: Release data source
Description

Stop maintaining config.js. The release workflow already generates and uploads a latest.json for the Tauri updater. Parse that file (or a variant of it) to build the download page, so the updater and the download site share one data source. This can be layered on top of any of the three code-placement options above.

Basis

direct: release.yaml uses tauri-apps/tauri-action@v1 with uploadUpdaterJson: true, so latest.json is already produced. external: Tauri updater JSON schema includes platform entries with URLs and signatures.

Rationale

It eliminates the manual drift between config.js and the actual release assets and turns a CI artifact into a shared source of truth. The download page becomes a consumer of the same manifest the updater already trusts.

Downsides

latest.json is designed for the updater, not humans, so it may lack friendly platform labels or file-size metadata; the schema could change; it may need augmentation for assets the updater does not consume (e.g., .rpm / .deb if not already included).

Rejection Summary

#IdeaReason rejected
1 Use the GitHub Release page as the canonical download destination Abandons the branded download page rather than combining it into the repo; does not meet the stated goal.
2 SaaS webhook-triggered download page rebuild Over-engineered; relies on external webhook complexity with no clear benefit over a release:published workflow trigger.
3 Docker-style CI-generated release manifest Merged into Idea 2 as a richer variant of the generated manifest approach; kept out to avoid near-duplication.
4 Browser-extension-store listing route Merged into Idea 1; the manifest-derived public route concept is already captured by the SvelteKit /download approach.
5 Deploy on release:published rather than tag push Merged into Ideas 1 and 2 as the recommended automation trigger; not a standalone recommendation.
